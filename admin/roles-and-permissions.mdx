Wraft uses Role-Based Access Control (RBAC) to govern what actions users can perform within an organisation. A role is a named set of permission strings scoped to a single organisation. A user can hold multiple roles simultaneously, and their effective permissions are the union of all permissions from all assigned roles.

---

## Role Schema

Each role record stores the following fields:

| Field | Type | Description |
|---|---|---|
| `name` | string | Role name, unique per organisation |
| `permissions` | array of strings | List of `resource:action` permission strings |
| `organisation_id` | UUID | Organisation this role belongs to |
| `user_count` | integer | Virtual field — number of users assigned this role |

---

## Default Roles

When a new organisation is created, Wraft automatically provisions two roles:

<CardGroup cols={2}>
  <Card title="superadmin" icon="crown">
    Has access to all permissions in the organisation. The creator of the organisation is automatically assigned this role. There must always be at least one superadmin — the last superadmin cannot be unassigned.
  </Card>
  <Card title="editor" icon="pen-nib">
    A pre-configured role with a standard set of document-editing permissions. Suitable for regular team members.
  </Card>
</CardGroup>

<Note>
  The `superadmin` role stores an empty `permissions` array in the database. At runtime, Wraft resolves it to the full set of all permissions defined in the system. You cannot delete the `superadmin` role.
</Note>

## Permission Model

Permissions are `resource:action` strings. Every API endpoint is guarded by a specific permission string that must be present in the authenticated user's effective permission set.

### Permissions Referenced in API Controllers

The following permission strings are declared in the Wraft API controllers and are used to guard the corresponding endpoints:

| Permission String | Guards |
|---|---|
| `role:show` | View roles (`GET /api/v1/roles`, `GET /api/v1/roles/:id`) |
| `role:manage` | Create, update, and assign roles |
| `role:delete` | Delete roles |
| `webhook:show` | View webhooks and logs |
| `webhook:manage` | Create, update, toggle, and test webhooks |
| `webhook:delete` | Delete webhooks |
| `api_key:show` | View API key metadata |
| `api_key:manage` | Create, update, and toggle API keys |
| `api_key:delete` | Delete API keys |
| `members:show` | View organisation members |
| `members:manage` | Invite and remove members |
| `workspace:update` | Update organisation settings |
| `workspace:delete` | Delete the organisation |

## Listing Roles

Retrieve all roles in the current organisation. Supports optional filtering by `name` and sorting by `name` or `name_desc`.

Requires the `role:show` permission.

```bash cURL
curl -X GET "https://your-wraft-instance.com/api/v1/roles?sort=name" \
  -H "Authorization: Bearer <your_jwt_token>"
```

```json Response
[
  {
    "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
    "name": "superadmin",
    "permissions": [],
    "organisation": { "id": "org-uuid", "name": "Acme Corporation" },
    "user_count": 2
  },
  {
    "id": "b2c3d4e5-f6a7-8901-bcde-f01234567891",
    "name": "editor",
    "permissions": ["document:manage", "document:show", "flow:manage"],
    "organisation": { "id": "org-uuid", "name": "Acme Corporation" },
    "user_count": 5
  }
]
```

## Creating a Custom Role

<Steps>
  <Step title="Determine the permissions the role needs">
    Choose from the permission strings listed in the table above. A role stores permissions as an array of strings.
  </Step>
  <Step title="POST to create the role">
    <CodeGroup>
    ```bash cURL
    curl -X POST https://your-wraft-instance.com/api/v1/roles \
      -H "Authorization: Bearer <your_jwt_token>" \
      -H "Content-Type: application/json" \
      -d '{
        "name": "document_reviewer",
        "permissions": [
          "document:show",
          "document:review",
          "flow:show"
        ]
      }'
    ```

    ```json Response
    {
      "id": "c3d4e5f6-a7b8-9012-cdef-012345678912",
      "name": "document_reviewer",
      "permissions": ["document:show", "document:review", "flow:show"],
      "organisation": {
        "id": "org-uuid",
        "name": "Acme Corporation"
      },
      "user_count": 0
    }
    ```
    </CodeGroup>

    Requires the `role:manage` permission.
  </Step>
</Steps>

## Updating a Role

Update a role's name or permissions array.

```bash cURL
curl -X PATCH https://your-wraft-instance.com/api/v1/roles/{id} \
  -H "Authorization: Bearer <your_jwt_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "senior_reviewer",
    "permissions": ["document:show", "document:review", "document:manage", "flow:show"]
  }'
```

Requires the `role:manage` permission.

## Assigning a Role to a User

```bash cURL
curl -X POST https://your-wraft-instance.com/api/v1/users/{user_id}/roles/{role_id} \
  -H "Authorization: Bearer <your_jwt_token>"
```

Wraft verifies that the target user is a member of the current organisation before assigning the role. After a successful assignment, the user receives an in-app notification.

Requires the `role:manage` permission.

---

## Unassigning a Role

```bash cURL
curl -X DELETE https://your-wraft-instance.com/api/v1/users/{user_id}/roles/{role_id} \
  -H "Authorization: Bearer <your_jwt_token>"
```

<Warning>
  Two safety rules apply when unassigning roles:
  1. You cannot remove the `superadmin` role if the user is the only superadmin in the organisation.
  2. Every member must hold at least one role — unassigning a user's only role is not permitted.
</Warning>

## Content Type Roles

Content Type Roles restrict which roles have access to create documents from a specific Content Type. This is useful when certain document types (e.g., contracts or legal agreements) should only be accessible to authorised roles.

### Assign a Role to a Content Type

```bash cURL
curl -X POST https://your-wraft-instance.com/api/v1/content_type_roles \
  -H "Authorization: Bearer <your_jwt_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "content_type_id": "ct-uuid-here",
    "role_id": "role-uuid-here"
  }'
```

### Remove a Role from a Content Type

```bash cURL
curl -X DELETE https://your-wraft-instance.com/api/v1/content_type_roles/{content_type_role_id} \
  -H "Authorization: Bearer <your_jwt_token>"
```
