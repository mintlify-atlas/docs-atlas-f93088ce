API keys provide a way for external systems, scripts, and server-to-server integrations to authenticate with the Wraft API without requiring a user session or JWT token. Unlike short-lived JWT access tokens, API keys are long-lived credentials that remain valid until explicitly deactivated or deleted. They are ideal for automated workflows, CI/CD pipelines, and backend services that need to interact with Wraft programmatically.

Each API key is scoped to one organisation and associated with the user who created it. When Wraft authenticates a request using an API key, it resolves the full set of permissions the associated user holds within that organisation — meaning the key inherits the creator's role-based permissions.

---

## Key Format

API keys generated by Wraft follow the format:

```
wraft_<8-char-prefix>_<32-char-random>
```

For example: `wraft_a1b2c3d4_XyZ1a2B3c4D5e6F7g8H9i0J1k2L3m4`

The prefix is stored in the database for fast lookup. The full key is hashed with bcrypt and **never stored in plaintext** — it is only returned once, immediately after creation.

## Creating an API Key

<Steps>
  <Step title="POST to create the key">
    <CodeGroup>
    ```bash cURL
    curl -X POST https://your-wraft-instance.com/api/v1/api_keys \
      -H "Authorization: Bearer <your_jwt_token>" \
      -H "Content-Type: application/json" \
      -d '{
        "name": "CI Pipeline Integration"
      }'
    ```

    ```json Response
    {
      "id": "ak-uuid-1234",
      "name": "CI Pipeline Integration",
      "key": "wraft_a1b2c3d4_XyZ1a2B3c4D5e6F7g8H9i0J1k2L3m4",
      "key_prefix": "a1b2c3d4",
      "is_active": true,
      "rate_limit": 1000,
      "ip_whitelist": [],
      "expires_at": null,
      "last_used_at": null,
      "usage_count": 0,
      "inserted_at": "2024-01-15T10:30:00Z",
      "updated_at": "2024-01-15T10:30:00Z"
    }
    ```
    </CodeGroup>

    Requires the `api_key:manage` permission.
  </Step>
  <Step title="Copy the key immediately — it will not be shown again">
    The `key` field in the response contains the full, unhashed API key. Wraft stores only a bcrypt hash of the key. Once you navigate away from the creation response, there is no way to retrieve the plaintext key again.

    <Warning>
      Store the key in a secrets manager (e.g., AWS Secrets Manager, HashiCorp Vault, or your CI/CD platform's secret store) immediately. If lost, you must delete the key and create a new one.
    </Warning>
  </Step>
</Steps>

### Request Body Fields

<ParamField body="name" type="string" required>
  A human-readable label for the key, between 3 and 100 characters. Must be unique within your organisation.
</ParamField>

<ParamField body="expires_at" type="string (ISO 8601)">
  Optional expiration date-time. If omitted, the key never expires. Must be a future datetime.
</ParamField>

<ParamField body="rate_limit" type="integer">
  Maximum number of requests per hour. Defaults to `1000`. Must be a positive integer.
</ParamField>

<ParamField body="ip_whitelist" type="array of strings">
  Optional list of IP addresses allowed to use this key. If empty (the default), all IPs are permitted.
</ParamField>

## Using an API Key

Pass the full key in the `x-api-key` header on every API request. No `Authorization: Bearer` header is needed when using an API key.

<CodeGroup>
```bash cURL
curl -X GET https://your-wraft-instance.com/api/v1/organisations/{id}/members \
  -H "x-api-key: wraft_a1b2c3d4_XyZ1a2B3c4D5e6F7g8H9i0J1k2L3m4"
```

```python Python
import requests

response = requests.get(
    "https://your-wraft-instance.com/api/v1/organisations/{id}/members",
    headers={
        "x-api-key": "wraft_a1b2c3d4_XyZ1a2B3c4D5e6F7g8H9i0J1k2L3m4"
    }
)
```

```javascript Node.js
const response = await fetch(
  "https://your-wraft-instance.com/api/v1/organisations/{id}/members",
  {
    headers: {
      "x-api-key": "wraft_a1b2c3d4_XyZ1a2B3c4D5e6F7g8H9i0J1k2L3m4"
    }
  }
);
```
</CodeGroup>

Wraft validates the key on every request by:
1. Extracting the 8-character prefix from the key
2. Querying all active keys with that prefix
3. Verifying the full key against each candidate's bcrypt hash
4. Checking the key is active, not expired, and the request IP is whitelisted (if configured)

Every successful authentication increments the key's `usage_count` and updates `last_used_at` asynchronously.

## Listing API Keys

Retrieve a paginated list of all API keys in the organisation. The response includes metadata only — the secret key value is never returned after creation.

```bash cURL
curl -X GET https://your-wraft-instance.com/api/v1/api_keys \
  -H "Authorization: Bearer <your_jwt_token>"
```

```json Response
{
  "api_keys": [
    {
      "id": "ak-uuid-1234",
      "name": "CI Pipeline Integration",
      "key_prefix": "a1b2c3d4",
      "is_active": true,
      "rate_limit": 1000,
      "ip_whitelist": [],
      "expires_at": null,
      "last_used_at": "2024-01-20T08:15:42Z",
      "usage_count": 342,
      "inserted_at": "2024-01-15T10:30:00Z"
    }
  ],
  "page_number": 1,
  "total_pages": 1,
  "total_entries": 1
}
```

Requires the `api_key:show` permission.

## Toggling a Key Active or Inactive

Disable a key to revoke its access immediately without deleting it. Call the endpoint again to re-enable the key.

```bash cURL
curl -X PATCH https://your-wraft-instance.com/api/v1/api_keys/{id}/toggle \
  -H "Authorization: Bearer <your_jwt_token>"
```

```json Response
{
  "id": "ak-uuid-1234",
  "name": "CI Pipeline Integration",
  "is_active": false,
  ...
}
```

Requires the `api_key:manage` permission.

<Tip>
  Toggling inactive rather than deleting is useful during incident response — you can disable a potentially compromised key instantly and re-enable it after investigation if it turns out to be a false alarm.
</Tip>

---

## Deleting an API Key

Permanently removes the key. This action is irreversible.

```bash cURL
curl -X DELETE https://your-wraft-instance.com/api/v1/api_keys/{id} \
  -H "Authorization: Bearer <your_jwt_token>"
```

Returns `204 No Content` on success. Requires the `api_key:delete` permission.

## Security Best Practices

<CardGroup cols={2}>
  <Card title="Minimum Permissions" icon="shield-halved">
    API keys inherit the creating user's role permissions. Create a dedicated user account with only the roles needed for the integration, then generate the key under that account.
  </Card>
  <Card title="Rotate Regularly" icon="rotate">
    Set an `expires_at` on every key and schedule rotation before expiry. Delete old keys immediately after issuing a replacement.
  </Card>
  <Card title="Never Commit to Source Control" icon="code-branch">
    Do not embed API keys in source code, configuration files, or Dockerfiles. Use environment variables or a dedicated secrets manager.
  </Card>
  <Card title="Use IP Whitelisting" icon="location-dot">
    If your integration originates from a fixed IP range (e.g., a cloud subnet), add the IPs to `ip_whitelist` so the key is rejected if used from unexpected origins.
  </Card>
  <Card title="Monitor Usage" icon="chart-line">
    Regularly check `last_used_at` and `usage_count` from the list endpoint. Unexpected spikes or access from unfamiliar times may indicate key compromise.
  </Card>
  <Card title="Use Rate Limits" icon="gauge-high">
    Set a `rate_limit` appropriate for the integration. This limits the blast radius if a key is ever leaked or misused.
  </Card>
</CardGroup>

<Warning>
  If you suspect an API key has been compromised, toggle it inactive immediately via `PATCH /api/v1/api_keys/:id/toggle`, investigate, and then delete it and create a replacement.
</Warning>
